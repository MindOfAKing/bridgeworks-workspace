# Oliviks Pre-Close Retrospective

Date: 2026-07-14
Status: Pre-close. Append the launch, acceptance, payment, and client-feedback outcomes before treating this as final.

## What Worked

### The replacement path reduced technical drag

The Next.js marketing site separates brand, menu discovery, and local proof from the existing WooCommerce transaction system. This preserved ecommerce scope while reducing the public page weight and simplifying the route to order.

### Client-owned systems were used

Menu content sits in the Oliviks Supabase project, MailerLite is under the client account, and WooCommerce remains the order system. This makes the handover more durable than a BridgeWorks-owned stack would be.

### A preview gate prevented an unauthorized launch

The new site remained on a Vercel preview while client approval and domain configuration were unresolved. That slowed the visible finish, but it protected the live business surface and made the remaining decision explicit.

### Evidence capture improved before cutover

Legacy and preview screenshots, route checks, data state, performance traces, scope controls, and technical decisions now exist before the old root is replaced. This will make the eventual case study more defensible.

## What Dragged

### Approval and ownership decisions arrived late

The final approver, menu and photo approval, address confirmation, incentive choice, and domain cutover were not resolved early enough to support a clean close.

### Optional and Foundation scope became blurred

The dedicated catering page exists even though it was an optional item in the proposal. Its commercial status must be decided before BridgeWorks describes it as contracted Foundation work.

### Operational systems were described ahead of proof

MailerLite is configured and valid, but the full signup, confirmation, and welcome path is untested. The WhatsApp pack and QR files exist, but the owner-created list, printing, and scan evidence remain open. Technical configuration is not the same as an accepted operating system.

### Public platform evidence is incomplete

Google Business Profile drafts, baseline work, and access exist, but final categories, attributes, Q&A, posts, menu, photos, NAP cleanup, and handover evidence are not complete in the current record.

### Domain and redirect planning should have started earlier

The Vercel warning on 8 July exposed unresolved root and `www` configuration. The old WordPress backup, obsolete-path redirects, and shop record protections should be a named workstream before launch week.

## Mistakes and Corrections

- Earlier handover language could sound more complete than the evidence allowed. The verification note now controls that wording.
- The final invoice was issued while several acceptance gates remained open. It must not be duplicated, and commercial status must stay separate from delivery status.
- The admin path uses a shared-secret workflow rather than owner authentication. It is acceptable only as an explicitly temporary handover mechanism.
- Schema includes approximate coordinates and a review count that require final verification. Syntax validation does not make those facts final.
- Full-page screenshots of animated sections can look blank unless the browser scrolls through the page. The audit now includes scroll-aware visual verification.

## What Changes for the Next Restaurant Build

1. Name the single approver and owner responsibilities at kickoff.
2. Lock the menu, prices, photos, rights, address, hours, and incentive before the main build sprint.
3. Create Search Console and analytics ownership at the start, not at handover.
4. Define the transaction boundary and redirect map before selecting the replacement architecture.
5. Capture a WordPress backup and baseline PageSpeed trace before implementation.
6. Run a written test matrix for forms, double opt-in, welcome delivery, WhatsApp, shop handoff, and mobile routes before issuing the balance invoice.
7. Treat configured, tested, live, handed over, and accepted as separate statuses.
8. Record optional goodwill work in writing so it does not become ambiguous billable scope.
9. Prepare the testimonial request and acceptance form before the walkthrough.
10. Capture production evidence immediately after launch while all routes and owners are available.

## Remaining Limitations

- The new site is not live on the approved root domain.
- Client approval of the final public facts and assets is not evidenced.
- Production form, retention, WhatsApp, and checkout handoffs are not fully tested.
- Search Console and analytics completion is not evidenced.
- Final GBP implementation evidence is incomplete.
- Owner walkthrough, acceptance, testimonial request, payment confirmation, and launch date remain open.
