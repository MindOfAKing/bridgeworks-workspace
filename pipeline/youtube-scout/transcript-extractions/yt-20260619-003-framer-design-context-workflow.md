## Core Thesis

Framer 3.0 turns website building into an agent-assisted design workflow. The strongest BridgeWorks takeaway is not to migrate all client sites to Framer. It is to create a design-context workflow where agents can use an existing codebase, `design.md`, content, and visual system to draft landing pages faster.

## Problem

AI-coded websites can be functional but visually weak. Design tools can look good but lose project context. Framer's agent and external-agent connection tries to bridge that gap.

## Workflow

1. Create or update `design.md` for the project.
2. Gather brand assets, site copy, screenshots, and offer notes.
3. Generate one landing-page direction in a design tool or local prototype.
4. Branch before visual experiments.
5. Use the agent for layout and section drafting.
6. Human-review copy, responsiveness, claims, and client scope.
7. Publish only after Emmanuel or client approval.

## Business Model

BridgeWorks can add a "landing page prototype sprint" for prospects or clients. Use it to show direction fast, then implement in the approved stack. This is especially useful for Build Brief, BridgeWorks landing pages, and small client campaign pages.

## Service Fulfillment Lessons

- Keep static-first Next.js/Vercel as the default for confirmed builds.
- Use Framer-style agent design as a prototype layer, not an automatic delivery stack.
- Create `design.md` for every serious website build.
- Do not let a fast prototype become unmanaged scope creep.
